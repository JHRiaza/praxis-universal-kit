## Memory Architecture (4+1 Levels)

PRAXIS organizes memory across 5 levels, from short-term to permanent:

| Level | Duration | Content | Where it lives |
|-------|----------|---------|---------------|
| L4.1 Operational | Current session | Active context, state | AI context window |
| L4.2 Episodic | Days–weeks | Sprint logs, decisions, corrections | `.praxis/metrics.jsonl` |
| L4.3 Semantic | Months–years | Lessons, rules, knowledge | SOUL.md, AGENTS.md |
| L4.4 Audit | Permanent | Full sprint history | Git history + JSONL |
| L4.5 Hardened | Permanent, verified | Facts with primary sources | This file |

---

## Level 1: Operational Memory

**What it is:** The AI's current context window — what it knows right now in this session.

**Best practices:**
- Load only what's needed for the current sprint
- Include: current task brief, relevant files, active constraints
- Don't overload: long contexts reduce attention quality
- At session start: explicitly load governance file + current project state

**Context loading checklist:**
```
□ Project purpose + current status
□ Active sprint brief
□ Relevant files/code
□ Current governance rules (this file + SOUL.md)
□ Known blockers or constraints
```

---

## Level 2: Episodic Memory

**What it is:** A record of what happened — the history of sprints and decisions.

**Stored in:** `.praxis/metrics.jsonl` (auto-maintained by `praxis log`)

**What gets recorded:**
- Task descriptions, durations, quality scores
- Model used, iteration count, interventions
- PRAXIS-Q scores (Phase B)
- Governance events

**How to use it:**
```bash
praxis status           # See aggregate stats
praxis export           # Get full JSONL for analysis
```

---

## Level 3: Semantic Memory

**What it is:** Accumulated knowledge — lessons learned, patterns, governance rules.

**Stored in:** SOUL.md (identity + principles) and AGENTS.md (procedures)

**Update rules:**
- Add new lessons after each sprint where something unexpected happened
- Update governance rules after incidents (see AGENTS.md)
- Review and prune outdated rules monthly

**Pattern to follow:**
```
[DATE] LESSON: [what was learned]
[DATE] RULE ADDED: [rule text] — REASON: [what incident prompted this]
[DATE] RULE UPDATED: [old rule → new rule] — REASON: [why changed]
```

---

## Level 4: Audit Memory

**What it is:** The permanent, immutable record of all decisions.

**Stored in:**
- Git history (code changes, file changes)
- `.praxis/metrics.jsonl` (sprint metrics)
- `.praxis/governance.jsonl` (governance events)

**Rule:** Once logged, audit records are never edited or deleted (except via `praxis withdraw`).

---

## Level 5: Hardened Facts

**What it is:** Facts that have been verified against primary sources.
Unlike lessons (which can be wrong), hardened facts have been checked.

**Trust scoring format:**
```
[FACT]: [statement]
[t:score | src:source | verified:date]
```

Where score is: `high` (directly verified), `medium` (strong secondary source), `low` (inferred)

## Memory Maintenance Schedule

| Frequency | Action |
|-----------|--------|
| After each sprint | `praxis log` — episodic memory auto-updated |
| After incidents | Update SOUL.md governance rules |
| Weekly | Review semantic memory for outdated rules |
| Monthly | Prune episodic logs, extract lessons to semantic |
| Per phase | `praxis export` before data collection ends |

---

## Memory Anti-patterns to Avoid

1. **Hallucinated facts as hardened facts** — Never add a fact to L4.5 without a primary source.
2. **Stale semantic memory** — Old rules that no longer apply create confusion.
3. **Missing episodic records** — Undocumented failures repeat themselves.
4. **Context overload** — Loading the entire codebase into every session reduces quality.
